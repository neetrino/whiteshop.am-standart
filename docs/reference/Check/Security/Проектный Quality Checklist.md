## A. Архитектура и границы модулей

> [!quote] Цель раздела
> обеспечить предсказуемую структуру, понятные границы ответственности и минимизировать “скрытые связи” между слоями. упростить сопровождение, тестирование и масштабирование (людьми, а не магией).

### 1. [ ] P0 — Ясные слои: UI → API → data-access → DB
- **Как проверить:** Просмотреть структуру `/app` или `/pages`, `/server`/`/api`, `/lib/db`
- **Как должно быть:** UI не делает SQL; API не лезет в JSX; доступ к БД только через data-access
- **Красные флаги:** SQL внутри компонентов, прямые запросы из route handlers без слоя
- **Что исправить:** Выделить репозитории/сервисы, вынести DB-код в `/lib/db/*`
- **Артефакт:** Диаграмма слоёв + ссылки на папки

### 2. [ ] P1 — Разделение server/client компонентов (Next.js)
- **Как проверить:** Поиск `use client`, импортов server-only
- **Как должно быть:** Client-компоненты не импортируют server-only модули
- **Красные флаги:** `fs`, `process.env` в client, утечки секретов
- **Что исправить:** Перенести логику на сервер, сделать API/Server Actions
- **Артефакт:** Линк на PR + список перемещённых файлов

### 3. [ ] P1 — Единый “composition root” для DI/инициализации
- **Как проверить:** Найти место инициализации клиентов (DB, S3/R2, Sentry)
- **Как должно быть:** Клиенты создаются централизованно и переиспользуются
- **Красные флаги:** Создание клиента на каждом запросе/рендере
- **Что исправить:** Вынести в `/lib/*/client.ts` с кешированием
- **Артефакт:** Ссылка на модуль инициализации

### 4. [ ] P1 — Явные границы доменов/модулей
- **Как проверить:** Проверить `/modules/*` или feature-based структуру
- **Как должно быть:** Feature-модули автономны, минимальные cross-imports
- **Красные флаги:** “Глобальный” `/utils` на 500 файлов
- **Что исправить:** Разделить по доменам, ввести публичные API модулей
- **Артефакт:** Мапа модулей (README)

### 5. [ ] P2 — Нет циклических зависимостей
- **Как проверить:** `madge`/`depcruise` (если настроено) или ручной аудит импортов
- **Как должно быть:** Циклов нет, слойность соблюдена
- **Красные флаги:** Тяжёлые импорты, “протаскивание” типов
- **Что исправить:** Разорвать циклы через интерфейсы/DTO
- **Артефакт:** Отчёт инструмента/скрин

### 6. [ ] P1 — Глобальные middleware и их ответственность
- **Как проверить:** Проверить `middleware.ts`
- **Как должно быть:** Middleware делает минимум: auth gating, redirects, headers
- **Красные флаги:** Middleware ходит в БД, делает тяжёлую логику
- **Что исправить:** Вынести логику в API/edge-safe решения
- **Артефакт:** Код middleware + объяснение

### 7. [ ] P2 — Feature flags и изоляция экспериментов
- **Как проверить:** Поиск `FEATURE_*`, конфиг флагов
- **Как должно быть:** Новые фичи защищены флагами при необходимости
- **Красные флаги:** Фича “в полёте” ломает прод
- **Что исправить:** Добавить флаг, дефолты, документировать
- **Артефакт:** Список флагов + где включается

### 8. [ ] P1 — Границы ошибок: где ловим исключения
- **Как проверить:** Проверить error boundaries, обработчики API
- **Как должно быть:** Ошибки локализованы, не валят весь рендер
- **Красные флаги:** “Uncaught” в логах, белые экраны
- **Что исправить:** Добавить error.tsx, обработчики ошибок API
- **Артефакт:** Скрин error UI + лог ошибки

### 9. [ ] P2 — Единый формат DTO/моделей на границе API
- **Как проверить:** Проверить схемы (zod/yup) и типы
- **Как должно быть:** DTO отдельно от ORM-типов, валидация на входе
- **Красные флаги:** Протекание DB-схемы наружу
- **Что исправить:** Ввести DTO + схемы валидации
- **Артефакт:** Примеры схем и типов

### 10. [ ] P1 — Стабильная стратегия кеширования (Next cache/revalidate)
- **Как проверить:** `fetch` options, `revalidateTag`, `revalidatePath`
- **Как должно быть:** Стратегия задокументирована и единообразна
- **Красные флаги:** Случайные `no-store` везде
- **Что исправить:** Описать правила, заменить на tag-based
- **Артефакт:** Таблица “что кешируем” в README

### 11. [ ] P2 — Версионирование API/контрактов (если нужно)
- **Как проверить:** Проверить `/api/v1/*` или заголовки
- **Как должно быть:** Изменения не ломают клиентов
- **Красные флаги:** “Тихие” breaking changes
- **Что исправить:** Добавить v1/v2, миграционный план
- **Артефакт:** Ссылка на OpenAPI/док

## B. Безопасность

> [!quote] Цель раздела
> предотвратить утечки данных, компрометацию аккаунтов и эксплуатацию типовых уязвимостей (OWASP). сделать безопасность “по умолчанию”, а не “по просьбе после инцидента”.

### 1. [ ] P0 — AuthN: единый механизм аутентификации
- **Как проверить:** Проверить auth middleware, провайдер (Auth.js/Clerk/custom)
- **Как должно быть:** Один источник истины, токены/сессии валидируются сервером
- **Красные флаги:** Разные механизмы в разных местах
- **Что исправить:** Унифицировать, удалить дубляжи
- **Артефакт:** Схема auth flow

### 2. [ ] P0 — AuthZ: проверки прав на каждом защищённом endpoint
- **Как проверить:** Просмотреть route handlers/API, сервисы
- **Как должно быть:** Проверка ролей/прав до доступа к данным
- **Красные флаги:** “Фронт спрятал кнопку” вместо серверной проверки
- **Что исправить:** Ввести guard/policy слой, unit-тесты
- **Артефакт:** Таблица ролей/разрешений

### 3. [ ] P0 — Защита сессий/куки (если cookie-based)
- **Как проверить:** Настройки cookies
- **Как должно быть:** `HttpOnly`, `Secure`, `SameSite=Lax/Strict`, корректный domain/path
- **Красные флаги:** `SameSite=None` без нужды, нет `HttpOnly`
- **Что исправить:** Исправить настройки, регресс-тест
- **Артефакт:** Скрин настроек/код

### 4. [ ] P0 — CSRF защита (если cookie auth + mutating запросы)
- **Как проверить:** Проверить CSRF токены/Origin checks
- **Как должно быть:** CSRF токен или строгая проверка `Origin`/`SameSite`
- **Красные флаги:** POST без CSRF, доверие `Referer`
- **Что исправить:** Включить CSRF middleware/проверки
- **Артефакт:** Тест: попытка CSRF

### 5. [ ] P0 — Валидация входных данных (schema validation)
- **Как проверить:** Поиск zod/yup/class-validator
- **Как должно быть:** Валидация на входе для body/query/params
- **Красные флаги:** Прямое использование `req.json()` без проверки
- **Что исправить:** Ввести схемы, единый error format
- **Артефакт:** Пример схемы + тест

### 6. [ ] P0 — Защита от SQL injection
- **Как проверить:** Проверить ORM/параметризацию
- **Как должно быть:** Только параметризованные запросы/ORM
- **Красные флаги:** Конкатенация SQL строк
- **Что исправить:** Заменить на параметризованные
- **Артефакт:** SAST/grep результаты

### 7. [ ] P0 — XSS защита и вывод пользовательского контента
- **Как проверить:** Проверить места `dangerouslySetInnerHTML`
- **Как должно быть:** Санитайзинг + CSP, безопасный рендер
- **Красные флаги:** Рендер HTML без очистки
- **Что исправить:** Ввести санитайзер, запретить опасные места
- **Артефакт:** Тест: XSS payload

### 8. [ ] P0 — CORS политика для API
- **Как проверить:** Настройки CORS/headers, Vercel/Render
- **Как должно быть:** CORS минимально необходимый, без `*` для приватного API
- **Красные флаги:** `Access-Control-Allow-Origin: *` с credentials
- **Что исправить:** Настроить allowlist доменов
- **Артефакт:** Скрин headers

### 9. [ ] P1 — Security headers (CSP, HSTS, X-Frame-Options, etc.)
- **Как проверить:** `next.config.js`, `headers()` на Vercel
- **Как должно быть:** Базовый набор заголовков включён
- **Красные флаги:** Нет CSP, нет HSTS на проде
- **Что исправить:** Добавить headers, протестировать
- **Артефакт:** Скрин response headers

### 10. [ ] P1 — CSP корректна и не “unsafe-everything”
- **Как проверить:** Проверить CSP
- **Как должно быть:** Минимум `unsafe-inline`, nonce/sha при необходимости
- **Красные флаги:** `script-src 'unsafe-inline' *`
- **Что исправить:** Ужесточить CSP по шагам
- **Артефакт:** Отчёт CSP evaluator

### 11. [ ] P0 — Rate limiting на auth/публичных endpoints
- **Как проверить:** Middleware/edge rate limit, Redis/Upstash
- **Как должно быть:** Лимиты на login, reset, public API
- **Красные флаги:** Брутфорс без ограничений
- **Что исправить:** Ввести лимиты + баны по IP/аккаунту
- **Артефакт:** Логи лимитов

### 12. [ ] P1 — WAF/бот-защита на perimeter (Cloudflare/Vercel)
- **Как проверить:** Настройки Cloudflare/WAF rules
- **Как должно быть:** Базовые правила включены, блок листы
- **Красные флаги:** Открытый публичный API без защиты
- **Что исправить:** Включить managed rules, firewall
- **Артефакт:** Скрин правил

### 13. [ ] P1 — Защита от SSRF (если есть загрузка URL/вебхуки)
- **Как проверить:** Поиск функций fetch по пользовательскому URL
- **Как должно быть:** Allowlist, запрет внутренних IP/метадаты
- **Красные флаги:** Можно дергать `169.254.169.254`
- **Что исправить:** Ввести фильтрацию/allowlist
- **Артефакт:** Тест SSRF

### 14. [ ] P1 — Upload security (типы файлов, размер, контент)
- **Как проверить:** Проверить загрузку файлов
- **Как должно быть:** Ограничения по MIME/size, проверка контента
- **Красные флаги:** Принимается “любое”
- **Что исправить:** Ввести проверки + скан
- **Артефакт:** Скрин конфигов

### 15. [ ] P1 — Secrets не попадают в клиент
- **Как проверить:** Поиск `process.env` в client, `NEXT_PUBLIC_*`
- **Как должно быть:** Только публичные переменные в клиенте
- **Красные флаги:** Секрет с префиксом `NEXT_PUBLIC`
- **Что исправить:** Переименовать/перенести на сервер
- **Артефакт:** Список env + аудит

### 16. [ ] P1 — Логи не содержат токены/PII
- **Как проверить:** Просмотр логирования, Sentry breadcrumbs
- **Как должно быть:** Маскирование чувствительных данных
- **Красные флаги:** JWT/пароли/телефоны в логах
- **Что исправить:** Маскирование + фильтры
- **Артефакт:** Пример “до/после”

### 17. [ ] P2 — Dependency scanning (npm audit/Snyk/GitHub)
- **Как проверить:** CI, GitHub alerts
- **Как должно быть:** Уязвимости отслеживаются и чинятся
- **Красные флаги:** Игнор зависимостей годами
- **Что исправить:** Включить alerts, обновить пакеты
- **Артефакт:** Отчёт сканера

### 18. [ ] P1 — Content security для embed/iframe
- **Как проверить:** Проверить `frame-ancestors`
- **Как должно быть:** Iframe запрещён или allowlist
- **Красные флаги:** Clickjacking возможен
- **Что исправить:** Добавить `frame-ancestors`/XFO
- **Артефакт:** Заголовки

### 19. [ ] P1 — Защита админки (доп. шаги)
- **Как проверить:** Проверить маршруты admin
- **Как должно быть:** Доп. пароль/2FA/allowlist по IP (если уместно)
- **Красные флаги:** Админка доступна всем после логина
- **Что исправить:** Усилить доступ, логировать действия
- **Артефакт:** Runbook админ доступа

### 20. [ ] P2 — Audit trail для критичных действий
- **Как проверить:** Проверить DB таблицы/лог
- **Как должно быть:** Логи действий (кто/что/когда)
- **Красные флаги:** “Никто не знает кто удалил”
- **Что исправить:** Добавить таблицу аудита/лог
- **Артефакт:** Пример записи аудита

### 21. [ ] P1 — Настроены лимиты и timeouts для API
- **Как проверить:** Vercel functions settings, Render server
- **Как должно быть:** Таймауты и лимиты соответствуют нагрузке
- **Красные флаги:** Висит бесконечно, OOM
- **Что исправить:** Ввести таймауты, ограничения
- **Артефакт:** Метрики latency

### 22. [ ] P2 — Security тесты/cheatsheets закреплены
- **Как проверить:** README/Runbook
- **Как должно быть:** Есть раздел “Security checks”
- **Красные флаги:** Знания только в голове
- **Что исправить:** Добавить чеклист и ссылки
- **Артефакт:** README section

## C. Управление секретами и env

> [!quote] Цель раздела
> исключить утечки секретов и обеспечить управляемость окружений (dev/preview/prod). минимизировать права и упростить ротацию.

### 1. [ ] P0 — Нет секретов в репозитории
- **Как проверить:** Поиск `grep -R "API_KEY\|SECRET"`; GitHub secret scanning
- **Как должно быть:** В репо только шаблон `.env.example`
- **Красные флаги:** `.env`, ключи в коде/README
- **Что исправить:** Удалить, ротация ключей
- **Артефакт:** Удалить, ротация ключей

### 2. [ ] P0 — Env scopes на Vercel (dev/preview/prod)
- **Как проверить:** Vercel Project → Settings → Env Vars
- **Как должно быть:** Чёткие значения по окружениям
- **Красные флаги:** Один и тот же секрет везде
- **Что исправить:** Разнести, документировать
- **Артефакт:** Скрин env scopes

### 3. [ ] P0 — Least privilege для Neon/R2/S3
- **Как проверить:** Проверить роли/ключи
- **Как должно быть:** Отдельные ключи per env, минимальные права
- **Красные флаги:** Один root ключ на всё
- **Что исправить:** Создать отдельные IAM/ролей
- **Артефакт:** Политики/roles docs

### 4. [ ] P1 — Ротация секретов и срок действия
- **Как проверить:** Runbook/политика
- **Как должно быть:** Есть процедура ротации
- **Красные флаги:** “Ключ с 2022 года живёт”
- **Что исправить:** Ввести календарь/процедуру
- **Артефакт:** Runbook

### 5. [ ] P1 — Проверка наличия env при старте
- **Как проверить:** Startup validation (zod/envsafe)
- **Как должно быть:** Приложение падает с понятной ошибкой, если env нет
- **Красные флаги:** Тихий `undefined` и баги
- **Что исправить:** Добавить env schema validation
- **Артефакт:** Лог старта

### 6. [ ] P1 — Разделение публичных и приватных env
- **Как проверить:** Аудит `NEXT_PUBLIC_*`
- **Как должно быть:** В клиенте только публичные значения
- **Красные флаги:** Секреты с NEXT_PUBLIC
- **Что исправить:** Переименовать/вынести
- **Артефакт:** Список env

### 7. [ ] P2 — Единый источник конфигурации
- **Как проверить:** `/config/*` или `env.ts`
- **Как должно быть:** Нет “размазанных” констант
- **Красные флаги:** Конфиг в 10 местах
- **Что исправить:** Централизовать config
- **Артефакт:** Ссылка на модуль

### 8. [ ] P1 — Render env и secret management
- **Как проверить:** Render dashboard/service settings
- **Как должно быть:** Env заданы на сервисе, не в коде
- **Красные флаги:** Env передаются через build args небезопасно
- **Что исправить:** Перенастроить Render env
- **Артефакт:** Скрин Render env

### 9. [ ] P2 — Ограничение доступа к env в команде
- **Как проверить:** IAM/доступы Vercel/Render
- **Как должно быть:** Только нужные роли видят секреты
- **Красные флаги:** Всем dev доступ к прод секретам
- **Что исправить:** Разделить роли, least access
- **Артефакт:** Список ролей

### 10. [ ] P2 — Локальный dev без прод-секретов
- **Как проверить:** `.env.local` с моками

# Diff-based Review (что проверять при маленьких изменениях)

* Изменён **endpoint** → проверить: AuthZ guard, schema validation, error format, rate limit, тесты (unit/integration).
* Изменён **запрос к БД** → проверить: `EXPLAIN`, индексы, N+1, транзакции, влияние на latency.
* Изменён **кеш/рендеринг** → проверить: `cache/no-store`, revalidate tags, регресс по LCP/CLS, корректность данных.
* Изменён **upload/storage** → проверить: presigned flow, IAM policy, CORS, content-type, size limits, приватность.
* Изменены **env/секреты** → проверить: scopes (preview/prod), отсутствие утечек в клиент, ротация при необходимости.
* Изменён **middleware/runtime** → проверить: edge-safe ограничения, совместимость с DB, latency, headers.
* Изменён **CI/CD** → проверить: обязательные checks, миграции, rollback, preview deploy.
