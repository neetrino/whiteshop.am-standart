___

## 1. Архитектурный уровень

### 1.1 Использование ORM / Query Builder

* [ ] Все запросы выполняются через ORM (Prisma / TypeORM / Knex и т.д.)
* [ ] Нет ручной конкатенации SQL-строк
* [ ] Не используется `queryRawUnsafe`, `executeRawUnsafe` или аналогичные небезопасные методы
* [ ] Все raw-запросы параметризованы

---

## 2. Параметризация запросов

### 2.1 Значения

* [ ] Ни один SQL-запрос не содержит `${userInput}` внутри строки
* [ ] Используются prepared statements (`?`, `$1`, `${param}` через ORM)
* [ ] Нет динамической вставки пользовательского ввода в `WHERE`, `INSERT`, `UPDATE`

### 2.2 Динамические фильтры

* [ ] Все фильтры строятся через безопасный API ORM
* [ ] Нет строковой сборки `WHERE ... AND ...` вручную

---

## 3. Опасные места (высокий риск)

Проверить вручную:

* [ ] `ORDER BY`
* [ ] `LIMIT / OFFSET`
* [ ] Названия колонок
* [ ] Названия таблиц
* [ ] Сложные отчёты / админ-поиск
* [ ] Экспорт данных
* [ ] Фильтры в CRM / Admin панелях

### 3.1 Whitelist

* [ ] Для сортировки используется whitelist допустимых колонок
* [ ] Направление сортировки строго `asc` или `desc`
* [ ] Нет возможности передать произвольное имя поля

---

## 4. Проверка кода (поиск по проекту)

Выполнить поиск:

* [ ] `SELECT `
* [ ] `INSERT `
* [ ] `UPDATE `
* [ ] `DELETE `
* [ ] `queryRaw`
* [ ] `executeRaw`
* [ ] `raw(`
* [ ] `${`
* [ ] `+ req.body`
* [ ] `+ req.query`

Если найдено:

* [ ] Проверить, используется ли параметризация
* [ ] Убедиться, что нет конкатенации строк

---

## 5. Валидация ввода

* [ ] Используется schema validation (Zod / class-validator / Joi)
* [ ] Ограничены типы данных (number, enum, string length)
* [ ] Нет “any” для входящих параметров
* [ ] Ограничена длина строк (например, до 255 символов)

Важно: валидация ≠ защита, но уменьшает поверхность атаки.

---

## 6. Права пользователя базы данных

* [ ] Приложение не использует superuser
* [ ] Нет прав `DROP DATABASE`
* [ ] Нет прав `CREATE EXTENSION`
* [ ] Пользователь имеет доступ только к нужным таблицам
* [ ] Отдельный пользователь для миграций

---

## 7. Ошибки и логирование

* [ ] Пользователю не показываются SQL-ошибки
* [ ] Stack trace скрыт в production
* [ ] Логи не содержат пароли и токены

---

## 8. Дополнительная защита

* [ ] Включён rate limiting
* [ ] Используется WAF (если есть)
* [ ] Включён dependency scanning
* [ ] Добавлены security-тесты

---

## 9. Ручной тест (проверка уязвимости)

Попробовать передать в формы:

```
' OR 1=1 --
' OR 'a'='a
1; DROP TABLE users;
```

Проверить:

* [ ] Запрос не выполняется некорректно
* [ ] Данные не раскрываются
* [ ] Ошибка корректно обработана

---
