> **👤 Вручную** — документ, панель провайдера или политики. Делаешь ты.

## 🤖 2.2 Secure cookies + session TTL/rotation

**Что сделать в коде (AI):**
- **Clerk:** в настройках клиента (или в Dashboard, см. 👤) проверить, что cookies ставятся с `HttpOnly`, `Secure`, `SameSite=Lax` (или `Strict`). В коде не переопределять на небезопасные.
- **Auth.js (NextAuth):** в конфиге `session.strategy`, `cookies`, `session.maxAge`, `session.updateAge`. Установить `cookieOptions: { httpOnly: true, secure: true, sameSite: 'lax' }`, настроить разумный `maxAge` и включить обновление сессии (updateAge) для ротации.

**Проверка:** В DevTools → Application → Cookies у сессионной куки стоят HttpOnly, Secure, SameSite. JS не может прочитать значение.

## 🤖 2.3 CSRF protection

**Что сделать в коде (AI):**
- Для cookie-based auth и любых мутирующих POST: проверять `Origin`/`Referer` на сервере (allowlist своих доменов) или использовать CSRF-токен (например в формах и API).
- Auth.js: при использовании Credentials provider и cookie — включить защиту от CSRF (по умолчанию через NextAuth часто учтено; для своих API добавить проверку вручную).

**Проверка:** POST с другого origin (или без валидного токена) возвращает 403 / invalid token.

---

## 🤖 2.4 RBAC server-side

**Что сделать в коде (AI):**
- В каждом защищённом route handler / Server Action / Nest guard: после проверки аутентификации проверять роль/права (например `user.role === 'admin'` или проверка по разрешению). Не полагаться на скрытие кнопок на клиенте.
- Централизовать проверки в одном месте (middleware, guard, wrapper), чтобы не забыть на новом endpoint.

**Проверка:** `curl` без токена или с токеном без прав на админ-действие возвращает 401/403, даже если в UI кнопка есть.

---

## 🤖 2.7 Logout invalidates session/token

**Что сделать в коде (AI):**
- **Clerk:** вызывать `signOut()` / правильный метод выхода; убедиться, что сессия инвалидируется на стороне Clerk (часто из коробки).
- **Auth.js:** при logout вызывать `signOut()` и при необходимости инвалидировать сессию в БД/хранилище. После выхода повторный запрос с тем же токеном/кукой должен получать 401.

**Проверка:** После logout повторный запрос с тем же cookie/token к защищённому API → 401 Unauthorized.

## 👤 2.5 Admin MFA mandatory

**Куда зайти → что сделать:**
- **Clerk:** **Configure → Multi-factor.** Включить MFA и настроить политику так, чтобы для роли/организации «admin» (или для выбранных пользователей) MFA было обязательно. Сохранить настройки.
- **Свой логин / Auth.js:** использовать провайдера с MFA или организационные меры; для P0 админов минимум — обязательная 2FA (например TOTP) и не пускать в админку без неё.

**Проверка:** Админ не может войти без прохождения MFA.

## 👤 2.6 Password policy (если не только OAuth)

**Что сделать вручную (Docs/Policy):**
- **Clerk:** **Configure → Email, Phone, Username** / настройки пароля: минимальная длина, сложность (если есть опции). Зафиксировать в политике/документе.
- **Свой логин:** в коде или в настройках провайдера задать минимальную длину и при необходимости проверку сложности; описать в документации.

**Проверка:** В настройках провайдера или в коде задана политика; в доке она описана.
