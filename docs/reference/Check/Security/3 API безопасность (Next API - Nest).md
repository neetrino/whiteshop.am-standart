# 3. API безопасность (Next API / Nest)

> [!info] Условные обозначения
> **🤖 В коде** — все пункты этого блока делаются в коде. Задаёшь AI: что реализовать в репозитории.

---

## 🤖 3.1 Input validation

**Что сделать в коде (AI):**
- Для всех API: body, query, params валидировать схемой (Zod в Next.js, class-validator + ValidationPipe в Nest).
- Не доверять `req.body`/`req.query` без проверки. При ошибке валидации возвращать 400 с понятным сообщением, не 500.

**Проверка:** Отправка мусорного payload (лишние поля, неверные типы) → 400 Bad Request, не 500.

---

## 🤖 3.1a XSS: sanitize user content

**Что сделать в коде (AI):**
- Не рендерить пользовательский ввод как HTML без санитизации. Избегать `dangerouslySetInnerHTML` с сырыми данными пользователя.
- Если нужен HTML — использовать санитайзер (например DOMPurify на клиенте или библиотека на сервере) и/или разрешать только безопасные теги.
- CSP (блок 1) дополнительно ограничивает выполнение скриптов.

**Проверка:** Ввод вида `<script>alert(1)</script>` или `<img onerror=...>` не выполняется и не отображается как исполняемый код.

---

## 🤖 3.2 No stack traces in prod

**Что сделать в коде (AI):**
- В production не отдавать в ответе клиенту `stack`, внутренние пути, детали ошибок. В Next.js: не показывать `NEXT_PUBLIC_*` с отладочной инфой; в обработчиках ошибок проверять `NODE_ENV === 'production'` и возвращать универсальное сообщение.
- В Vercel: убедиться, что деплой идёт с `NODE_ENV=production`.

**Проверка:** Специально вызвать 500 в prod — в теле ответа нет stack trace и путей к файлам.

---

## 🤖 3.2a No sensitive data in error responses

**Что сделать в коде (AI):**
- В сообщениях об ошибках (в т.ч. 4xx/5xx) не возвращать токены, пароли, внутренние ID БД, пути к файлам, детали схемы БД.
- Логировать детали на сервере для отладки, но в ответ клиенту отдавать только безопасный текст.

**Проверка:** При ошибке в ответе нет секретов и внутренних деталей.

---

## 🤖 3.3 Strict CORS

**Что сделать в коде (AI):**
- Настроить CORS с allowlist доменов (prod + при необходимости staging). Не использовать `Access-Control-Allow-Origin: *` для приватных API, особенно с credentials.
- В Next.js — в API route или в `next.config` headers; в Nest — настройки CORS в main/guard.

**Проверка:** Запрос с постороннего origin отклоняется или не получает нужных заголовков; с разрешённого — проходит.

---

## 🤖 3.4 Idempotency for critical POST

**Что сделать в коде (AI):**
- Для создания заказа, оплаты, обработки вебхуков: принимать заголовок `Idempotency-Key` (или аналог), сохранять в БД пару (key, результат/статус) и при повторе запроса с тем же ключом возвращать сохранённый результат без повторного списания/создания.
- Хранить ключи с TTL или по политике (например 24 часа).

**Проверка:** Два одинаковых запроса с одним Idempotency-Key → один заказ/одно списание, второй ответ без дублирования действия.

---

## 🤖 3.5 Webhook signature + replay protection

**Что сделать в коде (AI):**
- Проверять подпись вебхука (Stripe, провайдер и т.д.) по документации провайдера (HMAC, секрет из env). Не обрабатывать запросы с неверной подписью.
- Защита от повторов: проверять `timestamp` (отклонять слишком старые) и/или хранить обработанные id событий и не обрабатывать повторно.

**Проверка:** Поддельная подпись → 401/403. Повтор того же события с тем же id → не меняет состояние (идемпотентность).

---

## 🤖 3.6 Parameterized queries, no raw SQL with user input

**Что сделать в коде (AI):**
- Использовать только параметризованные запросы (Prisma, pg с параметрами, Knex с bindings). Никогда не подставлять пользовательский ввод в строку SQL через конкатенацию или шаблон.
- Если нужен raw query — только с плейсхолдерами (`$1`, `?`) и передачей значений массивом.

**Проверка:** Поиск по коду: нет конкатенации строки SQL с `req.body`, `req.query`, переменными из пользовательского ввода. SAST/grep не находит опасных шаблонов.
